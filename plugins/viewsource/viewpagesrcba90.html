The Signal-Slot system provides a means for realizing loosely coupled dependencies. In order to trigger a specific action when a certain event occurs, you must implement a Slot and register to listen to a specific Signal. The system then takes care, that the action is executed whenever the subscribed event occurs.

The system can execute an arbitrary number (0 to n) of Slots for each Signal. Each Slot must be crafted in a way, that it is unsignificant to it, if the Slot is executed immediately when the signalled action occurs or asynchronously at a later point in time.

Signal

A Signal represents a specific event, e.g. that a content version is published, and consists of information that is significant to the even, e.g. the content ID and version number. Therefore, a Signal is represented by an object of a specific class. The class name is used to identify each Signal uniquely. In order to get a Slot triggered by a Signal, it must be assigned to the corresponding class name.

Slot

A Slot is an extension of the system that is represented by a class that derives from eZ\Publish\Core\SignalSlot\Slot. The Slot must only implement a single method receive(), which is triggered by the system when a Signal for which the Slot is configured occurs. The receive() method receives the Signal object, to retrieve information from it. Since each Signal carries information that is specific to the event, it is highly recommended to implement a dedicated Slot for each event and for each Signal that is to be processed.

Example: Purging the Cache

On almost every update operation that occurs in the Core, parts of the Cache system need to be cleaned up. A perfect example for the use of Slots.
